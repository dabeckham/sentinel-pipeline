# Sentinel Pipeline — Disaster Recovery
*Keep this document updated as the system evolves.*

---

## What Needs to Be Protected

| Data | Location | Criticality | Recovery if lost |
|---|---|---|---|
| `.env` file | Docker host `~/sentinel-pipeline/.env` | **Critical** | All services fail to start; must regenerate secrets and reconfigure |
| PostgreSQL data | Docker volume `sentinel_postgres-data` | **Critical** | All job/detection/track metadata lost |
| MinIO snapshots | Docker volume `sentinel_minio-data` | **High** | All saved frame snapshots lost; source videos still on NAS |
| MinIO crops/frames | Docker volume `sentinel_minio-data` | **Medium** | Intermediate crops lost; can be regenerated by reprocessing |
| RabbitMQ state | Docker volume `sentinel_rabbitmq-data` | **Low** | In-flight jobs lost; files still on NAS and can be re-ingested |
| Source videos | NAS `/volume1/FTP/sentinel-ingest` | **High** | Managed by Synology — configure NAS backup separately |
| Code / config | GitHub | **Low** | Reclone from github.com/dabeckham/sentinel-pipeline |

## Backup Procedures

### 1. Back Up .env (do this immediately after first deploy)

The `.env` file contains all secrets and is NOT in git. Store it somewhere safe — a password manager, an encrypted USB drive, or a private note.

```bash
# Copy to a safe location (example: your home directory on another machine)
scp dabeckham@192.168.55.10:~/sentinel-pipeline/.env ~/sentinel-pipeline-env-backup.txt
```

**Required to restore:** JWT_SECRET_KEY, RABBITMQ_PASSWORD, POSTGRES_PASSWORD, MINIO_ACCESS_KEY, MINIO_SECRET_KEY

---

### 2. Back Up PostgreSQL

#### Manual backup
```bash
docker compose exec postgres pg_dump -U sentinel sentinel > backup_$(date +%Y%m%d_%H%M%S).sql
```

#### Restore from backup
```bash
docker compose exec -T postgres psql -U sentinel sentinel < backup_20260605_120000.sql
```

#### Automated daily backup (recommended — set up after Phase 1 deploy)
Add to crontab on the Docker host (`crontab -e`):
```bash
0 3 * * * cd ~/sentinel-pipeline && docker compose exec -T postgres pg_dump -U $POSTGRES_USER sentinel > ~/backups/sentinel_pg_$(date +\%Y\%m\%d).sql && find ~/backups -name "sentinel_pg_*.sql" -mtime +30 -delete
```
(Keeps 30 days of daily backups)

---

### 3. Back Up MinIO Snapshots

MinIO data lives in a Docker volume. Options:

#### Option A — Sync to NAS (recommended)
```bash
# Install MinIO client
wget https://dl.min.io/client/mc/release/linux-amd64/mc -O /usr/local/bin/mc && chmod +x /usr/local/bin/mc
mc alias set sentinel http://localhost:9000 $MINIO_ACCESS_KEY $MINIO_SECRET_KEY

# Sync snapshots to NAS
mc mirror sentinel/snapshots /mnt/ds-one/sentinel-snapshots-backup/
```

Add to crontab for nightly sync:
```bash
0 2 * * * mc mirror sentinel/snapshots /mnt/ds-one/sentinel-snapshots-backup/
```

#### Option B — Volume backup
```bash
docker run --rm \
  -v sentinel_minio-data:/data \
  -v ~/backups:/backup \
  alpine tar czf /backup/minio_$(date +%Y%m%d).tar.gz /data
```

## Recovery Procedures

### Scenario 1: Orchestrator container crashes / won't start

```bash
docker compose logs orchestrator          # check error
docker compose restart orchestrator       # simple restart
docker compose up -d --force-recreate orchestrator  # full recreate
```

Common causes:
- DB not ready yet → wait 30s and retry (healthcheck should prevent this)
- Missing env var → check `.env` has all required fields
- Migration failed → `docker compose exec orchestrator alembic current`

---

### Scenario 2: Full Docker host reboot

Everything should come back automatically on boot because:
- All containers have `restart: unless-stopped`
- NAS mount has `_netdev,nofail` in fstab (mounts before Docker starts)

Verify after reboot:
```bash
df -h | grep sentinel            # NAS mounted?
docker compose ps                # all containers up?
curl http://localhost:8000/api/health
```

---

### Scenario 3: Docker volume data loss (accidental `docker compose down -v`)

1. Restore PostgreSQL from latest backup:
```bash
docker compose up -d postgres
sleep 10
docker compose exec -T postgres psql -U sentinel sentinel < ~/backups/sentinel_pg_latest.sql
```

2. Restore MinIO snapshots from NAS backup:
```bash
docker compose up -d minio
sleep 10
mc mirror /mnt/ds-one/sentinel-snapshots-backup/ sentinel/snapshots
```

3. Start remaining services:
```bash
docker compose up -d
```

Note: jobs/detections metadata will be restored from the DB backup. Any detections processed after the last backup will need to be reprocessed from the source files on the NAS.

---

### Scenario 4: Complete host failure (new machine)

1. Set up new Docker host (install Docker, Docker Compose, nfs-common, NVIDIA Container Toolkit)
2. Mount NAS share (same fstab entry)
3. Clone repo: `git clone https://github.com/dabeckham/sentinel-pipeline.git`
4. Restore `.env` from your secure backup
5. Restore PostgreSQL backup (see Scenario 3)
6. Restore MinIO snapshots from NAS backup (see Scenario 3)
7. `docker compose up -d`

---

### Scenario 5: RabbitMQ data loss (in-flight jobs lost)

RabbitMQ queues are durable but if the volume is lost, in-flight jobs are gone. Recovery:

1. `docker compose restart rabbitmq` — queues are auto-created from `definitions.json` on startup
2. Jobs that were mid-processing will have status `md_processing` or `oc_processing` in the DB
3. After Phase 2: run the re-queue script to push stuck jobs back to the ingest queue:
```bash
docker compose exec orchestrator python -m app.tools.requeue_stuck_jobs
```
(This script will be written in Phase 5 hardening)

---

### Scenario 6: NAS unreachable

Orchestrator file watcher will log errors but not crash. When NAS comes back:
```bash
sudo mount -a    # remount if needed
```
Docker containers don't need restarting — the volume bind is live.

## Secret Rotation

If any secret is compromised:

### Rotate JWT secret
1. Update `JWT_SECRET_KEY` in `.env`
2. `docker compose restart orchestrator`
3. All active user sessions will be invalidated (users must log in again)

### Rotate DB password
```bash
docker compose exec postgres psql -U postgres -c "ALTER USER sentinel PASSWORD 'newpassword';"
# Update POSTGRES_PASSWORD in .env
docker compose restart orchestrator
```

### Rotate MinIO secret
1. Update in MinIO console (http://192.168.55.10:9001) → Identity → Service Accounts
2. Update `MINIO_SECRET_KEY` in `.env`
3. `docker compose restart orchestrator md-worker oc-worker`

## Health Check Summary

Run this after any recovery to confirm everything is working:

```bash
echo "=== Containers ===" && docker compose ps
echo "=== API ===" && curl -s http://localhost:8000/api/health | python3 -m json.tool
echo "=== NAS Mount ===" && df -h | grep sentinel
echo "=== GPU ===" && nvidia-smi --query-gpu=index,name,memory.used,memory.free --format=csv
echo "=== DB ===" && docker compose exec postgres pg_isready -U sentinel
```
